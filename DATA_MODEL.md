# 决策模型数据溯源

> **512GB or 1TB** — 面向开发者、设计师与创作者的存储容量决策工具。
> 本文档记录计算引擎中每一项数据的来源与依据，支持人工审查及 LLM 交叉验证。

---

## 1. 计算方法论

### 1.1 预估原则

**"模糊的准确，胜过精确的错误。"** 本工具不追求逐字节精确，而是帮助用户跨过"买 512GB 还是 1TB"的决策门槛。

### 1.2 单一预估值定义

每条工具/环境的体积采用**单一预估值**（GB），定义为：

> **正常使用约 1 年后的合理磁盘占用。**

隐含安装本体、用户缓存、增长膨胀三个因素，但不暴露为独立维度，避免追求不可观测的缓存精度。

### 1.3 工具层与项目层归属规则

为避免同一份数据在工具层和项目层重复计算，划分以下边界（经 GPT / GLM / Kimi 三方交叉验证确认）：

```
工具层 = 全局共享部分（二进制本体 + 全局缓存 + 全局插件 + 多版本残留）
项目层 = 单项目专属部分（node_modules / venv / target / .git / Library / 编译产物）
```

| 数据 | 归属 | 说明 |
|------|:--:|------|
| `node_modules` | 项目层 | npm 明确将依赖安装在项目目录内 |
| Git `.git` 目录、LFS 对象 | 项目层 | 跟随仓库，不属于全局工具 |
| Python 项目专属 venv | 项目层 | `venv/` 在项目内；conda base 保留在工具层 |
| Rust `target/` 编译目录 | 项目层 | 每个项目独立，不共享 |
| Unity `Library/` | 项目层 | 项目级资产缓存，与项目一一对应 |
| Xcode DerivedData | 工具层 | Xcode 统一管理，多个项目共享 |
| Gradle 全局缓存 | 工具层 | `~/.gradle/caches` 所有项目共享 |

### 1.4 来源可信度分级

| 等级 | 定义 | 示例 |
|:----:|------|------|
| **A** | 官方文档/商店页面明确标注的磁盘空间要求 | Steam 游戏页"需要 XX GB 可用空间" |
| **B** | 从官方渠道可推算的体积 | npm `unpackedSize`、GitHub Release 包体积 |
| **C** | 社区实测共识（多帖交叉验证）或实机测量 | 用户实测安装后文件夹大小 |
| **D** | 合理估算，标注"(估算)" | 无公开数据的冷门工具，基于同类软件推算 |

| 工具 | 实测安装体积 | 年损耗膨胀 | **预估值** | 来源 | 旧值 |
|------|:---:|:---:|:---:|:--:|:---:|

### 4.1 Design Assets（已删除）

原条目"Design Assets"（设计素材积累，50GB，D 级）已删除。跨项目复用的设计素材（图标、字体、模板、参考资源等）归入**设计师场景的单项目大小**中体现（8GB/项目，含约 3GB 素材引用缓冲）。

## 9. 状态判定与推荐容量

| 规则 | 定义 |
|------|------|
| 状态阈值 | 剩余 <10% → 紧张；10%-20% → 可用但需管理；20%-40% → 舒适；>40% → 充裕 |
| 推荐容量 | 总占用（含冗余）< 180GB → 256；180-400 → 512；400-850 → 1TB；>850 → 2TB |

> 产品规则，非技术数据。

## 10. 交叉验证指引

> 以下内容供投喂给其他 LLM 进行交叉验证时使用。

### 验证任务

你是一名存储容量估算专家。上方文档定义了一个新电脑存储需求计算工具的数据模型。请完成以下任务：

1. **逐条审查第 2-7 节中所有预估值。** 判断每个值是否合理。如果合理，回复"✅"。如果有争议，给出你的推荐值和依据。
2. **对标注 D 级的条目，给出你认为更合理的区间值**（如"3-8GB，中位 5GB"）。
3. **对第 7 节的单项目大小**，按场景给出你的经验判断——该场景下一个中位项目的典型磁盘占用。
4. **输出格式要求：**

```
| 条目 | 原文值 | 你的判断 | 推荐值或区间 | 依据简述 |
|------|--------|---------|-------------|---------|
```

### 回复时请注意

- 你是独立第三方，不需要同意原文。哪怕你判断"基本合理"，也请给出你的独立估算值以便交叉对比。
- 如果你的估算与原文偏差 >30%，请在依据中简要说明原因。
- 禁止说"取决于使用习惯"而不给数字——请给出一个合理的中位估计值及其区间。
